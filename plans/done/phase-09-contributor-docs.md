# Phase 9 — Contributor docs (component reference)

> **Read [`CONTEXT.md`](./CONTEXT.md) first.**

## Goal

A first-class contributor reference under `docs/contributing/` that documents every custom MDX component with copy-paste examples, frontmatter conventions, and a "how to add a page" walkthrough. The existing hidden `_dev/component-gallery.mdx` is for visual QA; this is for humans authoring content.

Modeled on `../TechReborn-Wiki-Upstream/docs/dev/wiki.mdx`, but using **our** components and conventions.

## Prerequisites

- Phases 0–8 complete.
- `docs/_dev/component-gallery.mdx` exists and renders all components correctly (use as the source of truth for current component APIs).
- All components in `src/components/` and registered in `src/theme/MDXComponents.tsx`.

## Steps

1. **Audit current component APIs.** Read each `src/components/*/index.tsx` and record:
   - Component name and its registered MDX tag (from `src/theme/MDXComponents.tsx`).
   - Required + optional props with types.
   - Realistic usage example pulled from somewhere in `docs/`.
   - Components in scope: `ItemIcon`, `Machine`, `RecipeFromData`, `MachineRecipeList`, `MultiblockShape`, `MachineStats`, `VersionContent` — plus any new components landed in later phases (10, 11) if they exist by the time this phase runs.

2. **Create `docs/contributing/index.mdx`** — landing page:
   - Frontmatter: `title: Contributing`, `sidebar_position: 99` (keep at bottom), `slug: /contributing`.
   - One paragraph orienting contributors: scope (TR for 1.20.1), licensing (MIT, NOTICE.md), the **never invent recipes** rule (lift from `CONTEXT.md`).
   - Links to the sub-pages built below.

3. **Create `docs/contributing/components.mdx`** — the component reference:
   - One H2 per component. Each section contains:
     - Short description (one or two sentences).
     - **Props table** (markdown table: name, type, default, description).
     - **Example** code block (MDX) followed by a live render of the same example so contributors see the output inline.
     - **When to use it** — one or two bullets distinguishing it from siblings (e.g. `ItemIcon` vs the new `ItemRef` from phase 10, `Machine` vs `RecipeFromData`).
   - At the end, a "Cheat sheet" subsection: a single MDX block showing the most common patterns side-by-side.

4. **Create `docs/contributing/authoring.mdx`** — page-authoring guide:
   - Required frontmatter fields and what each does (`title`, `sidebar_position`, `description`, `sidebar_class_name`).
   - Hero image convention from phase 11 (or a placeholder if phase 11 hasn't run — leave a `<!-- VERIFY phase 11: update once ItemHeader lands -->` comment).
   - Recipe section pattern: prefer `<RecipeFromData id="…"/>` over hand-typing recipes. Reiterate the never-invent rule.
   - Cross-linking conventions: use `[name](/docs/<path>)` for internal, `<ItemIcon id="…"/>` for icons, the new `<ItemRef>` (phase 10) for inline mentions with a clickable name.
   - "Adding a new machine page" walkthrough: copy the template, fill in stats from `src/data/`, drop `<MachineRecipeList machine="…"/>` at the bottom.
   - "Adding a new item page" walkthrough.

5. **Create `docs/contributing/data-pipeline.mdx`** — how the data is generated:
   - One paragraph each on `scripts/extract-tr-data.mjs`, `scripts/extract-vanilla-textures.mjs`, `scripts/build-items-with-pages.mjs`.
   - Re-run instructions: requires sibling clone of `../TechReborn` on branch `1.20.1`, run `./gradlew runDatagen` there first, then `npm run extract-data`.
   - What gets committed (`src/data/*.json`, `static/img/techreborn/*`, `static/img/vanilla/*`) and what doesn't.
   - Pointer to `CONTRIBUTING.md` at repo root for PR process.

6. **Wire up the sidebar.** In `sidebars.ts`, add a `cat("Contributing", "contributing")` entry at the bottom of `wikiSidebar`. The `index.mdx` provides the link target; the autogenerated children pull in `components`, `authoring`, `data-pipeline`.

7. **Cross-link from `CONTRIBUTING.md`** at repo root: add a "See also: [`/docs/contributing`](https://jakehowden.github.io/TechRebornWiki/docs/contributing) for component and page-authoring docs." line.

1. `git mv plans/phase-09-contributor-docs.md plans/done/phase-09-contributor-docs.md`

5. Report: deployed `/docs/contributing` URL, any prop discrepancies between components and their TS types.

## On failure

- A component's prop API is unclear or undocumented in TS → record it in the audit step output, surface as a P2 issue, and document the prop you observed in practice. Don't block this phase on a typing cleanup.
- Sidebar entry collides with another phase's edits → resolve in the merge, keep "Contributing" at the very bottom.
