# Phase 24 — Coverage audit + cleanup

> **Read [`CONTEXT.md`](./CONTEXT.md) first.**

## Goal

After Phases 20, 21, 22, 23, every `<ItemRef>` and `<ItemIcon>` in the wiki should resolve to a real page (or a deliberate, documented external fallback). This phase confirms that and cleans up the long tail.

The end state: a user can click any item icon anywhere on the wiki and land on a relevant page — no dead ends.

## Prerequisites

- Phases 20, 21, 22, 23 complete (or at least Phases 20 + 21 + 23 — Phase 22 is independent UX work).

## Steps

1. **Run the coverage audit.**
   ```sh
   node scripts/audit-coverage.mjs
   ```
   Capture before/after numbers vs. the audit reports from Phases 19/20/21. The "before" baseline should already be in those phase reports — quote them in the new commit body.

2. **For any remaining unmapped items, decide per-item:**
   - **Add a stub page** (preferred) — extend Phase 21's generator if a category-wide gap remains, or hand-author a page for one-offs.
   - **Document as deliberately external** — e.g. obscure vanilla items that fall through to `https://minecraft.wiki/...`. Add to a new `docs/contributing/coverage.mdx` section so future agents understand why they're not in our index.

3. **Verify there are no broken `<ItemRef>` resolutions caused by tag indirection.** Tags (`#c:silicon_plates`, etc.) resolve to a representative item id; that id must have a page. Quick check:
   ```js
   // node scripts/audit-tag-coverage.mjs (new, ~15 lines)
   const tags = require('../src/data/tags.json');
   const pages = require('../docs/items-with-pages.json');
   for (const [tag, id] of Object.entries(tags)) {
     if (!pages[id]) console.log(`tag ${tag} → ${id} has no page`);
   }
   ```
   Each line of output is a navigation gap. Add the missing items to the appropriate generator family or add a stub page.

4. **Sidebar review.** With ~500 generated pages, ensure:
   - Each `_category_.json` is `collapsed: true`.
   - The sidebar isn't unreadable when fully expanded.
   - Family folders with >40 children are split or hand-curated. Top suspects: `docs/items/` (catch-all) and `docs/blocks/misc-generated/`.
   - The "Materials" top-level entry shows per-family collapsibles (Dusts, Ingots, Plates, Parts, Gems, etc.) without leaf duplication. Reuse the `category-index-hidden` pattern from `sidebars.ts` if needed.

5. **Run `npm run build`** with `onBrokenLinks: "throw"`. Fix every warning. Common culprits:
   - `<RecipeFromData id="...">` with an id that doesn't exist in `recipes.json` after a data refresh.
   - `<ItemRef>` to an id whose page route changed during Phase 20 / 21 generation.

6. **Run `npm run typecheck`.** Should already be clean — confirm.

7. **Manual smoke test.** `npm start`, then walk these paths:
   - `materials/dusts.mdx` → click an `<ItemRef>` to a part → arrive on a Phase 20 part page.
   - `processing/hv-machines/distillation-tower.mdx` → verify Phase 22 fluid labels.
   - Any page with `<ItemIcon id="minecraft:furnace" />` → verify Phase 23 3D sprite.
   - `materials/gems.mdx` → click `Sapphire` → arrive on the Phase 21 world-obtained Sapphire page.
   - Random walk: pick 5 random items from `items.json` and verify each has a working link from at least one upstream page.

8. **Update `docs/contributing/coverage.mdx`** (create if absent) with:
   - Current coverage % (mapped items / total items).
   - List of intentionally-external items with rationale.
   - Quick guide for future agents on how to add a new item to coverage.

## Verification

- [ ] Coverage audit: ≥99% of items have a page or a documented external fallback.
- [ ] `node scripts/audit-tag-coverage.mjs` reports zero unresolvable tags.
- [ ] `npm run build` passes with `onBrokenLinks: "throw"`.
- [ ] `npm run typecheck` passes.
- [ ] No console warnings during `npm start` for missing textures or broken refs.
- [ ] Manual smoke test passes for each path in step 7.
- [ ] `docs/contributing/coverage.mdx` exists and is current.

## On failure

- Coverage stuck below 99% because a category has zero recipes AND zero downstream uses → those items are genuinely orphaned in the data. Either author a brief manual page (preferred) or document the orphan in `coverage.mdx` and accept the gap.
- Sidebar still feels overwhelming after collapses → consider hand-curating `sidebars.ts` for the worst sections rather than relying on autogenerated children. Hand-curated wins over autogenerated when count is high.
- Build fails on a broken link from a page generated in an earlier phase → don't patch the symptom in this phase; go back and fix the generator. Otherwise the bug recurs on the next regeneration.
- Tag coverage check finds many gaps → batch-fix by adding the missing ids to a Phase 20 / 21 generator family rather than authoring 50 individual stubs.
